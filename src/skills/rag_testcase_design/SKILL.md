# RAG测试用例设计专家技能（数据优先版）

## 技能概述

本技能定义了基于RAG（检索增强生成）的测试用例设计专家的行为规范，**核心原则是优先使用原始数据，而非LLM生成的内容**。

## 核心理念：数据驱动测试设计

### 为什么优先原始数据？

| 对比维度 | 原始数据 (rag_query_data) | LLM生成 (rag_query) |
|---------|--------------------------|-------------------|
| **准确性** | 直接来自知识库，100%可靠 | 可能产生幻觉，需要验证 |
| **可追溯性** | 可定位到具体文档和章节 | 难以追溯来源 |
| **细节完整** | 包含所有字段定义和规则 | 可能遗漏关键细节 |
| **一致性** | 与系统实现保持一致 | 可能与实际有偏差 |
| **适用场景** | 精确的测试数据、状态规则 | 概念理解、流程概述 |

### 数据驱动的测试设计流程

```
┌─────────────────┐
│  原始数据检索    │  ← rag_query_data + rag_graph_get
│  (实体/关系/规则) │
└────────┬────────┘
         ▼
┌─────────────────┐
│  数据分析整理    │  ← 提取状态机、字段规则、业务流程
│  (结构化处理)    │
└────────┬────────┘
         ▼
┌─────────────────┐
│  测试场景推导    │  ← 从实体属性、关系、规则推导
│  (数据驱动)      │
└────────┬────────┘
         ▼
┌─────────────────┐
│  测试用例生成    │  ← 每条用例都有数据依据
│  (可追溯验证)    │
└─────────────────┘
```

## RAG工具使用优先级

### 🔴 第一优先：rag_query_data（结构化数据检索）

**用途**：获取知识库中的**原始实体、关系、文本块**

**输出内容**：
```json
{
  "entities": [
    {
      "entity_name": "现付订单",
      "content": "订单的一种类型，支付时机为先付款后发货...",
      "source": "order_system_spec.md"
    }
  ],
  "relations": [
    {
      "src_id": "现付订单",
      "tgt_id": "工单",
      "content": "现付订单支付完成后产生工单",
      "source": "process_flow.md"
    }
  ],
  "chunks": [
    {
      "content": "预收抵扣规则：处理完工单后产生应收，优先使用预收进行抵扣...",
      "source": "financial_rules.md",
      "chunk_index": 5
    }
  ]
}
```

**推荐参数**：
```python
{
  "mode": "hybrid",           # 混合检索，兼顾精确和召回
  "top_k": 100,               # 获取更多实体
  "chunk_top_k": 10,          # 获取更多文本块
  "include_chunk_content": True  # 必须包含完整内容
}
```

---

### 🟡 第二优先：rag_graph_get（知识图谱子图获取）

**用途**：获取指定实体周围的**关联实体和关系网络**

**使用流程**：
```
rag_graph_search("订单") 
→ 找到精确实体名称 "现付订单"
→ rag_graph_get(entity_name="现付订单", max_depth=2)
→ 获取完整的实体关系图
```

---

### 🟢 第三优先：rag_graph_search（知识图谱实体搜索）

**用途**：**模糊搜索**知识图谱中的实体标签

**适用场景**：
- 不确定实体精确名称（如"订单"还是"销售单"）
- 探索知识库中有哪些相关实体

---

### ⚪ 最后选择：rag_query（LLM生成式查询）

**用途**：获取**概念性解释**和**流程概述**

**使用限制**：
- ⚠️ **禁止**作为测试用例的唯一依据
- ⚠️ 必须与 `rag_query_data` 结果**交叉验证**
- ✅ 仅用于理解术语定义
- ✅ 用于补充原始数据不足时的上下文

---

## 标准测试用例模板

基于业务场景测试的标准格式，每条用例必须包含以下字段：

| 字段 | 说明 | 必填 | 数据来源 |
|------|------|------|---------|
| 业务场景 | 一句话描述测试的核心业务场景 | ✅ | 实体关系定义 |
| 前置条件 | 执行测试前必须满足的条件和数据状态 | ✅ | 实体状态定义 |
| 测试步骤&流程 | 按序号列出的详细操作步骤 | ✅ | 业务流程文本块 |
| 预期结果 | 每个步骤对应的具体验证点 | ✅ | 规则文本块 |
| 测试数据 | 具体使用的测试数据（金额、数量、状态等） | ✅ | 字段定义规则 |
| 数据来源 | 引用原始数据的出处 | 建议 | 检索结果索引 |

## 数据驱动的测试场景设计

### 从实体定义推导测试数据

**示例**：订单实体定义
```json
{
  "entity_name": "现付订单",
  "content": "...",
  "properties": {
    "order_amount": {
      "type": "Decimal(10,2)",
      "min": 0.01,
      "max": 99999999.99,
      "required": true
    },
    "status": {
      "type": "enum",
      "values": ["待支付", "已支付", "处理中", "已完成", "已取消"]
    }
  }
}
```

**推导的测试数据**：
- 边界值：0.01、99999999.99
- 异常值：0、-1、99999999.999（小数位溢出）
- 状态组合：所有状态的有效流转

### 从关系定义推导测试场景

**示例**：实体关系
```json
{
  "src_id": "现付订单",
  "tgt_id": "工单",
  "relation_type": "产生",
  "trigger_condition": "订单状态 = 已支付",
  "content": "现付订单支付完成后，系统自动产生对应的工单"
}
```

**推导的测试场景**：
- 正常场景：订单已支付 → 工单正常产生
- 异常场景：订单未支付 → 不应产生工单
- 边界场景：支付中（状态未确定）→ 如何处理

### 从规则定义推导验证点

**示例**：业务规则
```json
{
  "chunk": "预收抵扣规则：处理完工单后，系统产生应收。应收金额优先使用客户预收进行抵扣，抵扣后剩余部分为客户应付。",
  "source": "financial_rules.md"
}
```

**推导的验证点**：
- 预收 > 应收：抵扣全部应收，剩余预收不变
- 预收 < 应收：抵扣全部预收，产生剩余应收
- 预收 = 应收：全部抵扣，无剩余

## 标准工作流程

### Phase 1: 原始数据检索（必须先执行）

```
Step 1: 使用 rag_query_data 检索业务实体定义
        查询: "[业务实体]的定义、字段、规则"
        参数: top_k=100, chunk_top_k=10, include_chunk_content=true

Step 2: 使用 rag_query_data 检索业务流程和规则
        查询: "[业务场景]的流程、规则、状态流转"
        参数: mode="hybrid"

Step 3: 使用 rag_graph_search 定位关键实体
        查询: "订单、工单、售后单"

Step 4: 使用 rag_graph_get 获取实体关联图
        参数: entity_name="现付订单", max_depth=2

Step 5: 整理数据结构
        - 实体清单（名称、属性、约束）
        - 状态机（状态定义、转换条件）
        - 关系映射（触发关系、数据依赖）
        - 规则清单（计算公式、校验规则）
```

### Phase 2: 测试场景分析

基于原始数据，分析：

```
1. 实体属性分析 → 测试数据设计（边界值、异常值）
2. 状态机分析 → 状态流转测试（所有有效路径）
3. 关系分析 → 流程测试（触发条件、异常分支）
4. 规则分析 → 验证点设计（计算逻辑、校验规则）
5. 组合分析 → 集成场景（多实体协同）
```

### Phase 3: 测试用例生成

按照标准模板生成用例，确保：

1. **数据来源标注**：每条用例标注数据出处
2. **可追溯验证**：预期结果能在原始数据中找到依据
3. **完整性覆盖**：覆盖原始数据中识别的所有场景

## 测试用例质量规范

### 用例编号格式
```
TC-[业务域]-[场景缩写]-[3位序号]

示例：
- TC-ORDER-PREPAY-001 (现付订单场景)
- TC-WORK-REFUND-001 (工单退款场景)
- TC-AFTER-RETURN-001 (售后退货场景)
```

### 优先级定义
| 级别 | 名称 | 场景描述 |
|------|------|---------|
| **P0** | 阻塞级 | 核心业务流程，失败则阻塞发布 |
| **P1** | 高优先级 | 重要功能，影响主要用户路径 |
| **P2** | 中优先级 | 常规功能，覆盖正常场景 |
| **P3** | 低优先级 | 边缘场景、优化类 |

### 预期结果书写规范（基于原始数据）

**合格示例**（基于规则文本块）：
```
规则来源: financial_rules.md
预期结果:
  1. 预收1000后查看客户预收退款页面有1000的预存
  2. 仅退款200后，查看预收退费200，预收剩800
  3. 处理完工单后，销售单已完成，产生1000的应收，折扣200，抵扣800的预收
```

**禁止出现**：
- ❌ "页面正常显示" → 必须描述具体显示内容
- ❌ "操作成功" → 必须描述成功的具体表现
- ❌ "数据正确" → 必须描述数据的具体状态

## 业务场景设计原则（数据驱动）

### 现付订单场景矩阵（从实体关系推导）

| 维度 | 取值 | 数据来源 |
|------|------|---------|
| 支付时机 | 先付后单 / 先单后付 | 订单类型定义 |
| 订单完成 | 未完成 / 部分完成 / 已完成 | 状态机定义 |
| 售后操作 | 仅退款 / 退货退款 / 换货 | 售后类型定义 |
| 工单状态 | 未处理 / 处理中 / 已完成 / 已终止 | 工单状态定义 |
| 资金流转 | 预收 / 应收 / 退款 / 抵扣 | 财务规则定义 |

## 质量自检清单（数据验证）

每完成一组用例，执行以下检查：

- [ ] **数据来源验证**：每条用例的预期结果都能在原始数据中找到出处
- [ ] **状态准确性**：所有状态流转都符合知识图谱中的状态机定义
- [ ] **数据一致性**：涉及多实体的用例，数据关系与知识图谱一致
- [ ] **规则验证**：数值计算、业务规则与原始文本块描述一致
- [ ] **覆盖度检查**：所有从原始数据中识别出的场景都有用例覆盖
- [ ] **边界覆盖**：测试数据覆盖了字段定义中的边界条件
- [ ] **异常场景**：包含原始数据中提及的异常处理规则

## 数据不足时的处理

当原始数据不足以支撑测试设计时：

1. **明确告知数据缺失**：
   ```
   ⚠️ 数据缺失警告
   - 未检索到"退款规则"的明确定义
   - 未检索到"换货差价计算"的具体公式
   - 建议：补充知识库文档或确认规则
   ```

2. **基于假设的设计**（标注为待确认）：
   ```
   基于以下假设生成用例（需用户确认）：
   [假设1]: 退款金额不能大于订单金额
   [假设2]: 退款后订单状态变为"已退款"
   ```

3. **提供数据检索建议**：
   ```
   建议补充以下文档：
   - refund_policy.md（退款政策）
   - exchange_rules.md（换货规则）
   ```
